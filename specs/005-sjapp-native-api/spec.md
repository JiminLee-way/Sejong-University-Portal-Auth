# Feature Specification: sjapp 네이티브 REST API 전면 전환

**Feature Branch**: `005-sjapp-native-api`
**Created**: 2026-03-19
**Status**: Draft
**Input**: sjpt WebSquare를 완전히 제거하고 sjapp.sejong.ac.kr의 네이티브 REST API만 사용하여 전체 시스템 재구축

## User Scenarios & Testing

### User Story 1 - 성적 조회 (Priority: P1)

개발자가 학생의 전체 기이수 성적, 당학기 성적, 특정 학기 성적을 JSON API로 조회한다. 기존 WebSquare HTML 파싱 방식 대신 구조화된 JSON 응답을 직접 받아 사용한다.

**Why this priority**: 성적 조회는 가장 핵심적인 기능이며, 기존 시스템에서 세션 충돌 문제가 가장 빈번하게 발생한 영역이다.

**Independent Test**: 로그인 후 전체 성적 조회를 호출하면 학기별 성적과 GPA를 포함한 JSON을 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 유효한 계정, **When** 전체 성적 조회를 요청하면, **Then** 모든 학기의 과목별 성적과 전체 GPA를 반환한다
2. **Given** 유효한 계정, **When** 당학기 성적 조회를 요청하면, **Then** 현재 학기의 성적만 반환한다
3. **Given** 유효한 계정과 특정 연도/학기, **When** 학기별 성적 조회를 요청하면, **Then** 해당 학기의 성적을 반환한다
4. **Given** 잘못된 비밀번호, **When** 로그인을 시도하면, **Then** 인증 실패 오류를 반환한다

---

### User Story 2 - 공지사항 조회 (Priority: P1)

개발자가 세종대 공지사항을 8개 카테고리별로 목록 조회하고, 개별 공지의 상세 내용을 가져온다. HTML 파싱 없이 JSON API로 직접 조회한다.

**Why this priority**: 공지사항은 학생 서비스의 기본 기능이며, 기존 HTML 파싱 방식보다 JSON API가 훨씬 안정적이다.

**Independent Test**: 학사공지 목록을 요청하면 제목, 작성일, 작성자를 포함한 목록을 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 카테고리와 페이지 지정, **When** 공지사항 목록을 요청하면, **Then** 해당 카테고리의 공지 목록(제목, 작성일, 작성자, 조회수)을 반환한다
2. **Given** 공지 ID와 카테고리, **When** 공지 상세를 요청하면, **Then** 본문 내용을 포함한 상세 정보를 반환한다
3. **Given** 카테고리와 개수 지정, **When** 최신 공지를 요청하면, **Then** 해당 카테고리의 최신 N개 공지를 반환한다

**8개 카테고리**: 일반공지, 학사공지, 장학공지, 취업/창업공지, 국제교류공지, 채용/모집공지, 공학교육센터공지, 학술정보원공지

---

### User Story 3 - 세종뉴스 및 피드 (Priority: P2)

개발자가 세종대의 뉴스(교내뉴스, 언론속세종, 보도자료, 웹진)와 소셜 피드(블로그, 유튜브)를 조회한다.

**Why this priority**: 뉴스와 피드는 대학 정보 서비스의 부가 기능으로, 공지사항보다 활용 빈도가 낮다.

**Independent Test**: 세종뉴스 목록을 요청하면 뉴스 제목, 날짜, 요약을 포함한 목록을 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 뉴스 유형 지정, **When** 뉴스 목록을 요청하면, **Then** 해당 유형의 뉴스 목록(제목, 날짜, 요약, 썸네일)을 반환한다
2. **Given** 피드 유형 지정, **When** 피드 목록을 요청하면, **Then** 블로그/유튜브 피드(제목, 링크, 출처, 날짜)를 반환한다

**뉴스 5종**: 교내뉴스, 언론속세종, 보도자료, 세종웹진, 공학웹진
**피드 3종**: 최신, 블로그, 유튜브

---

### User Story 4 - 일정 관리 (Priority: P2)

개발자가 학생의 개인 일정을 생성, 조회, 수정, 삭제(CRUD)할 수 있다. 카테고리, 태그, 리마인더, 메모 기능을 포함한다.

**Why this priority**: 일정 관리는 개인화 기능으로, 학사 데이터 조회보다 활용 범위가 좁다.

**Independent Test**: 일정을 생성한 후 목록에서 확인하고, 수정/삭제가 정상 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자, **When** 일정을 생성하면, **Then** 생성된 일정이 목록에 나타난다
2. **Given** 기존 일정이 있을 때, **When** 일정을 수정하면, **Then** 변경 내용이 반영된다
3. **Given** 기존 일정이 있을 때, **When** 일정을 삭제하면, **Then** 목록에서 사라진다
4. **Given** 로그인된 사용자, **When** 반복 일정을 생성하면, **Then** 반복 패턴에 따라 일정이 생성된다

---

### User Story 5 - 문의(QnA) (Priority: P2)

개발자가 학생의 1:1 문의를 생성, 조회, 수정, 삭제할 수 있다. 오류 신고, 기능 요청 등 카테고리별로 관리한다.

**Why this priority**: 문의 기능은 학생 지원 서비스의 일부로, 핵심 학사 기능보다 우선순위가 낮다.

**Independent Test**: 문의를 작성한 후 내 문의 목록에서 확인되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자, **When** 문의를 작성하면, **Then** 내 문의 목록에 나타난다
2. **Given** 문의 카테고리 목록을 요청하면, **Then** 사용 가능한 카테고리(오류 신고, 기능 요청 등)를 반환한다

---

### User Story 6 - 교직원 검색 (Priority: P3)

개발자가 세종대 교직원을 이름, 학과, 전화번호, 이메일로 검색한다.

**Why this priority**: 교직원 검색은 보조 기능이다.

**Independent Test**: 교직원 목록을 요청하면 이름, 학과, 연락처를 포함한 결과를 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자, **When** 교직원 검색을 요청하면, **Then** 이름, 학과, 전화번호, 이메일을 포함한 목록을 반환한다

---

### User Story 7 - 알림(UMS) (Priority: P3)

개발자가 학생의 알림 수신함을 조회하고, 읽음 처리, 삭제, 알림 설정을 관리한다.

**Why this priority**: 알림은 실시간 서비스의 보조 기능이다.

**Independent Test**: 수신함 목록과 읽지 않은 알림 수를 조회하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인된 사용자, **When** 수신함을 조회하면, **Then** 알림 목록(내용, 날짜, 읽음 여부)을 반환한다
2. **Given** 읽지 않은 알림이 있을 때, **When** 읽지 않은 알림 수를 조회하면, **Then** 정확한 카운트를 반환한다
3. **Given** 로그인된 사용자, **When** 알림 설정을 조회/변경하면, **Then** 카테고리별 알림 수신 여부가 반영된다

---

### User Story 8 - 인증 관리 (Priority: P1)

개발자가 로그인, 로그아웃, 토큰 갱신, 내 정보 조회를 수행한다. 모든 인증 기능은 JWT 기반으로 동작한다.

**Why this priority**: 인증은 모든 기능의 기반이다.

**Independent Test**: 로그인 → 내 정보 조회 → 토큰 갱신 → 로그아웃 흐름이 정상 동작하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 유효한 학번/비밀번호, **When** 로그인하면, **Then** 접근 토큰과 사용자 정보(이름, 학과, 학년, 역할)를 반환한다
2. **Given** 유효한 토큰, **When** 내 정보를 조회하면, **Then** 사용자 상세 정보를 반환한다
3. **Given** 유효한 토큰, **When** 토큰 갱신을 요청하면, **Then** 새 접근 토큰을 반환한다
4. **Given** 유효한 토큰, **When** 로그아웃하면, **Then** 토큰이 무효화된다

---

### Edge Cases

- 토큰이 만료된 상태에서 보호된 API를 호출하는 경우
- 동시에 여러 클라이언트가 같은 계정으로 로그인하는 경우
- sjapp 서버가 점검 중인 경우
- 페이지네이션에서 범위를 초과한 페이지를 요청하는 경우
- 존재하지 않는 공지 ID로 상세 조회를 요청하는 경우
- 빈 검색 결과가 반환되는 경우
- 알림 대량 삭제/읽음 처리 시 일부만 성공하는 경우

## Requirements

### Functional Requirements

- **FR-001**: 시스템은 학번/비밀번호로 로그인하여 접근 토큰을 획득할 수 있어야 한다
- **FR-002**: 시스템은 토큰 갱신을 통해 세션을 연장할 수 있어야 한다
- **FR-003**: 시스템은 전체/당학기/학기별 성적을 조회할 수 있어야 한다
- **FR-004**: 시스템은 8개 카테고리별 공지사항 목록을 페이지네이션으로 조회할 수 있어야 한다
- **FR-005**: 시스템은 개별 공지사항의 상세 내용을 조회할 수 있어야 한다
- **FR-006**: 시스템은 5종의 세종뉴스 목록과 상세를 조회할 수 있어야 한다
- **FR-007**: 시스템은 3종의 소셜 피드(최신, 블로그, 유튜브)를 조회할 수 있어야 한다
- **FR-008**: 시스템은 개인 일정의 생성, 조회, 수정, 삭제를 수행할 수 있어야 한다
- **FR-009**: 시스템은 반복 일정, 카테고리, 태그, 리마인더, 메모를 관리할 수 있어야 한다
- **FR-010**: 시스템은 1:1 문의의 생성, 조회, 수정, 삭제를 수행할 수 있어야 한다
- **FR-011**: 시스템은 교직원을 검색할 수 있어야 한다
- **FR-012**: 시스템은 알림 수신함 조회, 읽음 처리, 삭제를 수행할 수 있어야 한다
- **FR-013**: 시스템은 알림 설정(카테고리별 수신 여부)을 조회/변경할 수 있어야 한다
- **FR-014**: 시스템은 내 정보(이름, 학과, 학년, 역할, 카드번호)를 조회할 수 있어야 한다
- **FR-015**: 모든 응답은 구조화된 JSON 형식이어야 하며, HTML 파싱에 의존하지 않아야 한다
- **FR-016**: 인증이 필요 없는 공개 API(공지, 뉴스, 피드)는 토큰 없이도 호출 가능해야 한다
- **FR-017**: 기존 도서관 좌석 기능(libseat)은 그대로 유지해야 한다

### Key Entities

- **AuthToken**: 인증 토큰 — 접근 토큰, 토큰 유형, 만료 시간, 사용자 정보
- **UserProfile**: 사용자 프로필 — 학번, 이름, 학과, 학년, 역할, 카드번호, 이메일
- **GradeReport**: 성적 보고서 — 학기별 성적 목록, 전체 GPA, 취득 학점
- **Notice**: 공지사항 — ID, 카테고리, 제목, 작성자, 작성일, 조회수, 본문
- **News**: 세종뉴스 — ID, 카테고리, 제목, 요약, 작성일, 조회수, 썸네일
- **Feed**: 소셜 피드 — ID, 출처(블로그/유튜브), 제목, 링크, 날짜, 설명
- **Schedule**: 일정 — ID, 제목, 시작/종료 시간, 카테고리, 태그, 리마인더, 메모
- **QnA**: 문의 — ID, 카테고리, 제목, 내용, 상태
- **Staff**: 교직원 — ID, 이름, 학과, 전화번호, 이메일
- **Notification**: 알림 — ID, 내용, 날짜, 카테고리, 읽음 여부

## Success Criteria

### Measurable Outcomes

- **SC-001**: 모든 API 응답이 구조화된 JSON 형식으로, HTML 파싱 없이 데이터를 조회할 수 있다
- **SC-002**: 성적 조회가 2초 이내에 완료된다 (기존 5초 대비 60% 단축)
- **SC-003**: 브라우저 포털 로그인 중에도 모든 API가 세션 충돌 없이 동작한다
- **SC-004**: 공지사항 8개 카테고리 모두 목록 + 상세 조회가 정상 동작한다
- **SC-005**: 일정/문의의 CRUD 전체 흐름이 정상 동작한다
- **SC-006**: 토큰 만료 시 갱신을 통해 재로그인 없이 서비스를 계속 사용할 수 있다
- **SC-007**: 기존 도서관 좌석 기능이 변경 없이 유지된다
- **SC-008**: 인증 불필요한 공개 API는 토큰 없이 호출 가능하다

## Assumptions

- sjapp.sejong.ac.kr의 모든 API는 안정적으로 운영되며, 모바일 앱과 동일한 데이터를 제공한다
- JWT 접근 토큰의 만료 시간은 30분(1800초)이며, 갱신 토큰으로 연장 가능하다
- 공개 API(publicapi)는 인증 없이 호출 가능하며, 보호 API(secureapi)는 Bearer 토큰이 필요하다
- sjapp의 성적 API는 sjpt WebSquare 성적 API와 동일한 데이터를 반환하지만, 필드 구조가 다를 수 있다
- 기존 도서관 좌석 기능(libseat)은 sjapp에 없으므로 기존 구현을 유지한다
- sjapp API의 페이지네이션은 0-indexed page + size 방식이다
- sjapp API의 에러 응답은 `{ status: "error", code: "...", message: "..." }` 형식이다
