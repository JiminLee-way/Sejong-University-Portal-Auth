# Feature Specification: sjapp JWT 모바일 인증 전환

**Feature Branch**: `004-sjapp-jwt-auth`
**Created**: 2026-03-19
**Status**: Draft
**Input**: 기존 portal.sejong.ac.kr SSO 로그인을 sjapp.sejong.ac.kr JWT 기반 모바일 인증으로 전환

## User Scenarios & Testing

### User Story 1 - 브라우저 세션 충돌 없는 인증 (Priority: P1)

개발자가 학생 계정으로 로그인하여 학사 데이터를 조회할 때, 학생이 동시에 브라우저에서 포털에 접속해 있더라도 세션 충돌 없이 정상 동작한다. 기존에는 "동일한 사용자가 이미 프로그램을 사용하고 있으므로 자동 종료됩니다" 오류가 발생했지만, 모바일 앱과 동일한 인증 방식으로 전환하면 독립적인 세션이 생성되어 충돌이 발생하지 않는다.

**Why this priority**: 현재 시스템의 가장 큰 문제점인 세션 충돌을 해결하는 핵심 변경.

**Independent Test**: 브라우저에서 포털에 로그인한 상태에서 라이브러리로 성적 조회를 호출하여 정상 응답을 받는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 학생이 브라우저에서 포털에 로그인된 상태, **When** 라이브러리로 성적 조회를 요청하면, **Then** 세션 충돌 없이 성적 데이터를 정상 반환한다
2. **Given** 유효한 학번/비밀번호, **When** 로그인 후 성적 조회를 요청하면, **Then** 5초 이내에 성적 데이터를 반환한다
3. **Given** 잘못된 비밀번호, **When** 로그인을 시도하면, **Then** 명확한 인증 실패 오류를 반환한다

---

### User Story 2 - 기존 API 호환성 유지 (Priority: P1)

개발자가 기존에 사용하던 모든 API(성적, 수강내역, 장학이력, 시간표, 등록금, 졸업요건, 학사일정, 도서관, 공지사항)가 인증 방식 변경 후에도 동일한 인터페이스와 응답 형식으로 동작한다.

**Why this priority**: 기존 사용자(개발자)의 코드가 깨지지 않아야 하며, 인증 방식 변경이 API 소비자에게 투명해야 한다.

**Independent Test**: 기존 테스트 코드를 변경 없이 실행하여 모든 API가 동일한 응답을 반환하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 새 인증 방식으로 로그인한 클라이언트, **When** 기존과 동일한 방식으로 모든 조회 API를 호출하면, **Then** 동일한 형식의 응답을 반환한다
2. **Given** 새 인증 방식, **When** REST API 서버를 통해 동일한 엔드포인트로 요청하면, **Then** 기존과 동일한 JSON 응답을 반환한다

---

### User Story 3 - 연속 API 호출 안정성 (Priority: P2)

개발자가 한 번 로그인한 후 여러 API를 순차적으로 호출할 때, 페이지 전환 시 세션이 끊기거나 오류가 발생하지 않는다.

**Why this priority**: 실제 사용 시 여러 API를 순차 호출하는 패턴이 일반적이며, 이 안정성이 보장되어야 실용적이다.

**Independent Test**: 로그인 후 성적→수강→장학→시간표를 연속 호출하여 모두 정상 응답하는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 로그인된 클라이언트, **When** 성적, 수강, 장학, 시간표를 연속으로 호출하면, **Then** 모든 호출이 세션 오류 없이 정상 응답한다
2. **Given** 로그인된 클라이언트, **When** 포털 API와 도서관 API를 교대로 호출하면, **Then** 두 시스템 모두 정상 동작한다

---

### Edge Cases

- 모바일 인증 서버(sjapp)가 점검 중이지만 포털(sjpt)은 정상인 경우
- 인증 토큰이 만료된 후 데이터 조회를 시도하는 경우
- 동일 계정으로 여러 클라이언트가 동시에 인증하는 경우
- SSO 리다이렉트 URL 발급 후 포털이 일시적으로 응답하지 않는 경우
- 학번/비밀번호가 모바일 인증 서버에 등록되지 않은 경우

## Requirements

### Functional Requirements

- **FR-001**: 시스템은 모바일 인증 서버를 통해 학번/비밀번호로 인증할 수 있어야 한다
- **FR-002**: 시스템은 인증 후 발급된 토큰으로 포털 세션을 자동 생성할 수 있어야 한다
- **FR-003**: 시스템은 브라우저 포털 세션과 독립적으로 동작하여 세션 충돌이 발생하지 않아야 한다
- **FR-004**: 시스템은 기존 모든 API의 인터페이스를 변경 없이 유지해야 한다
- **FR-005**: 시스템은 한 번 로그인 후 여러 API를 순차적으로 호출할 때 세션을 안정적으로 유지해야 한다
- **FR-006**: 시스템은 인증 실패 시 기존과 동일한 오류 타입을 반환해야 한다
- **FR-007**: 시스템은 모바일 인증 서버 접속 불가 시 명확한 오류를 반환해야 한다

## Success Criteria

### Measurable Outcomes

- **SC-001**: 브라우저에서 포털 로그인 중인 상태에서 모든 API 호출이 세션 충돌 없이 정상 동작한다
- **SC-002**: 기존 API의 요청/응답 형식이 100% 동일하게 유지된다
- **SC-003**: 로그인 후 4개 이상의 API 연속 호출이 세션 오류 없이 완료된다
- **SC-004**: 로그인에서 데이터 반환까지 5초 이내에 완료된다
- **SC-005**: 잘못된 비밀번호 입력 시 2초 이내에 인증 실패 오류를 반환한다

## Assumptions

- sjapp.sejong.ac.kr 모바일 인증 서버는 portal.sejong.ac.kr과 동일한 학번/비밀번호 체계를 사용한다
- 모바일 인증 서버에서 발급된 SSO 토큰으로 sjpt.sejong.ac.kr에 접근하면 브라우저 세션과 독립적인 별도 세션이 생성된다
- 모바일 인증 서버는 학기 중 안정적으로 운영되며 포털과 독립적으로 가용하다
- SSO 리다이렉트 토큰은 1회성이며 발급 후 짧은 시간 내에 사용해야 한다
- 기존 세션 저장/복원 메커니즘은 새 인증 방식에서도 동일하게 활용 가능하다
